# 2. Do It Again

> 🌐 [English](../02-do-it-again.md) · **日本語**

**Q1.** ここに配列があります: `[1, 2, 3]`。各要素を2倍にして言ってみてください。

**A1.** `2, 4, 6`。いま頭の中で `map` をしましたね。

---

**Q2.** では Align で。`[1, 2, 3].map(fn x { x * 2 }).sum()` は何ですか？

**A2.** `12`。`map` が各要素を2倍にし — `[2, 4, 6]` — `sum` がそれらを足しました。

---

**Q3.** `fn x { x * 2 }` は何ですか？

**A3.** ラムダ（lambda）です。名前を持たない関数です。`fn`、引数、ブロック。引数の型は、流れ込んでくる要素から推論されます。

---

**Q4.** 名前つきの関数を代わりに使えましたか？

**A4.** はい。使えます:

```align
fn double(x: i64) -> i64 = x * 2

[1, 2, 3].map(double).sum()
```

答えは同じ `12`。シグネチャさえ合えば、パイプラインのステージにはどんな関数でも渡せます。

---

**Q5.** `[1, 2, 3].map(fn x { x * x }).sum()` は何ですか？

**A5.** `14`。各要素を2乗した結果の和 `1 + 4 + 9` となります。

---

**Q6.** `[].map(fn x { x * x }).sum()` — 何もないものを map したら？

**A6.** ひっかけ問題ですね。空の配列リテラルには、要素の型を推論する手がかりがありません。しかし、もし `i64` の空の *スライス* を `map` したなら、合計は `0` になります。無を2乗しても、無のままだからです。

---

**Q7.** lambda は、自分の外にある名前を使えますか？

```align
factor := 3
[1, 2, 3].map(fn x { x * factor }).sum()
```

**A7.** はい — 答えは `18` です。lambda は `factor` を値で**キャプチャ**します。作られた瞬間にコピーを取るのです。

---

**Q8.** もし `factor` が `mut` で、`map` の *あと* に変わったら、map はそれを見ますか？

**A8.** いいえ。**値で** capture されるということは copy されるということです。変化を共有する環境はありません。仕事を明示的に並列化するとき、これは特に重要になります。lambda は変化する `factor` を共有しません。

---

**Q9.** `[1, 2, 3].map(fn x { x * 2 })` — `.sum()` なしでは何ですか？

**A9.** コンパイルエラーになります。パイプラインは終端処理（畳み込みか実体化）によって終わらせる必要があります。評価されない途中のままのパイプラインは隠れたコストになりますが、Align はそうした隠しごとを許容しません。

---

**Q10.** では、2倍にした配列を本当に *取っておきたい* なら、どうしますか？

**A10.** その場合は明示的に指示します: `[1, 2, 3].map(fn x { x * 2 }).to_array()`。これで `[2, 4, 6]` が手に入り、メモリ確保（アロケーション）の存在がコード上で明確になります。

---

**Q11.** `[1, 2, 3].map(fn x { x * 2 }).map(fn x { x + 1 }).sum()` は何ですか？

**A11.** `15`。map を2つ続けて: `[3, 5, 7]`。そしてここが秘密です — コンパイラは**ループを1つだけ**作ります。2つではありません。ステージは融合(fuse)されます。

---

**Q12.** Q11 は途中でいくつ配列を作りましたか？

**A12.** ゼロです。`3`、`5`、`7` はそれぞれ一瞬だけレジスタに宿り、すぐさま足し込まれました。だからこそ、途中経過を取っておくのではなく、pipeline を終わらせるよう要求されるのです。

---

**Q13.** 逐次 `map` の中で `print` してもよいですか？

```align
[1, 2].map(fn x {
    print(x)
    x * 2
}).sum()
```

**A13.** はい。`1`、`2` の順に表示し、答えは `6` です。逐次 stage は副作用を持てます。到達した各要素について、入力順・stage 順にちょうど一度実行されます。

---

**Q14.** それでも pipeline の中で表示するのが、たいてい悪い習慣なのはなぜ？

**A14.** 答えと観測を混ぜ、最適化を制限し、同じ関数を `par_map` で使えなくするからです。自然に Pure にできる stage は Pure にしましょう。ただし神話ではなく規則を覚えてください。逐次 `map` は副作用を許し、明示的な並列処理は許しません。

---

**Q15.** `map` が `par_map` になると、何が変わりますか？

**A15.** worker の順序は観測できる物語ではなくなるため、callable は Pure でなければなりません。表示も外部状態の変更も不可です。並列性は source に書かれ、それを安全にする規律も source に現れます。

---

**Q16.** 各要素から指を離さずに追ってください。

```align
[2, 4, 6]
    .map(fn x { x + 1 })
    .map(fn x { x * x })
    .to_array()
```

**A16.** `[9, 25, 49]`。一つの要素が chain 全体を通ります。`2 → 3 → 9`、次に `4 → 5 → 25`、そして `6 → 7 → 49`。

---

**Q17.** 2つの map を入れ替えると、同じ答え？

**A17.** いいえ。`[5, 17, 37]`。今度は `2 → 4 → 5` です。どちらも全要素を訪ねますが、関数の合成には順序があります。

---

**Q18.** すべての score に10を足し、新しい score を取っておく。終端は？

**A18.**

```align
scores.map(fn s { s + 10 }).to_array()
```

`.to_array()` です。要求は多数の score を返すことなので、多数を materialize します。

---

**Q19.** すべての score に10を足し、合計だけを求める。終端は？

**A19.**

```align
scores.map(fn s { s + 10 }).sum()
```

`.sum()` です。要求は一つの答えなので、直接 collapse します。一つへ向かう途中で、多数を作ってはいけません。

---

**Q20.** `offset` は `7`。「すべての point を右へ7移動する」を書いてください。

**A20.**

```align
offset := 7
moved := points.map(fn x { x + offset }).to_array()
```

lambda は、必要な外側の事実を一つ携えます。

---

**Q21.** 同じ変換を5か所で使います。何を変えるべき？

**A21.** 名前をつけます。

```align
fn move_right(x: i64) -> i64 = x + 7
```

そして `.map(move_right)` とします。inline lambda は局所的な考え、名前つき関数は繰り返す考えのためにあります。

---

> **第二の戒律**
>
> *各要素を訪ねたいなら、`map` せよ。結果を取っておきたいなら、`to_array()` と声に出して言え。*

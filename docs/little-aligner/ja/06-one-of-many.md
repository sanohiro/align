# 6. One of Many

> 🌐 [English](../06-one-of-many.md) · **日本語**

**Q1.** 信号は赤か黄か緑です。同時に2つになることはありますか？

**A1.** いいえ。多くのうちの一つ — それが sum type です:

```align
Light { Red, Yellow, Green }
```

---

**Q2.** 緑のものはどう作りますか？

**A2.** `Light.Green`。いつでもその型で修飾してください — 修飾なしの `Green` では、それがどの型のものか分からないからです。

---

**Q3.** いまどれを持っているかは、どう尋ねますか？

**A3.** 尋ねません — `match` します:

```align
fn go(l: Light) -> i64 = match l {
    Red    => 0,
    Yellow => 0,
    Green  => 1,
}
```

---

**Q4.** アームの中で、なぜ `Light.Red` ではなく `Red` なのですか？

**A4.** `match` の中では、対象（scrutinee）の型がすでに分かっています。そのため、アームでは修飾なしのバリアント名をそのまま使えます。ただし外側のコードで値を作る時は、修飾が必要です。

---

**Q5.** `Green` のアームを消したら、どうなりますか？

**A5.** コンパイルエラーになります。`match` がすべてのバリアントを網羅しなくなったからです。**網羅性が契約です。** 来年バリアントを増やせば、プログラム中のすべての `match` がエラーとして教えてくれます。

---

**Q6.** `Red` と `Yellow` は答えを共有します。アームを2つ書かねばなりませんか？

**A6.** 1つのアームが両方を持てます: `Red | Yellow => 0`。あるいは残りをまとめて処理します: `_ => 0` — ただし使う場合は自己責任です。`_` は、将来追加されるかもしれないバリアントまで飲み込みます。

---

**Q7.** バリアントは何かを運べますか？

```align
Shape { Circle(i64), Rect(i64, i64), Dot }
```

**A7.** はい — ペイロードを保持できます。`Shape.Circle(10)` は半径を保持した円です。`Shape.Dot` は何も保持しません。

---

**Q8.** 他の言語なら、クラスの継承と仮想メソッド `draw()` を使って様々な図形を扱うところです。なぜ Align では直和型と `match` を使うのですか？

**A8.** 継承はロジックを多数のファイルに分散させます。さらに動的ディスパッチ（仮想メソッド）は、呼び出し先を間接ジャンプの向こう側に隠します — 予測しづらく、インライン化もベクトル化もできません。直和型と `match` はロジックを1箇所に集め、すべてのケースが処理されていることをコンパイル時に保証します。CPUは見通しの良い一本道を好み、読み手は真実のすべてが1箇所に書かれていることを好むのです。

---

**Q9.** ペイロードはどうやって取り出しますか？

**A9.** `match` のアームで変数名を割り当てます:

```align
fn area(s: Shape) -> i64 = match s {
    Circle(r)  => 3 * r * r,
    Rect(w, h) => w * h,
    Dot        => 0,
}
```

`area(Shape.Rect(3, 4))` は `12`。ペイロードは、パターンマッチで取り出したアームの中でだけ存在します — そもそも `Rect` かもしれない値に対して `.radius` を直接呼ぶことはできません。

---

**Q10.** `match` は文ですか、式ですか？

**A10.** 式です。Q9 の関数本体まるごとが式そのものです。変数に束縛したり、戻り値にしたり、引数として渡したりできます: `verdict := match s { ... }`。

---

**Q11.** 数に対して match してもよいですか？ `match n { 0 => ..., _ => ... }`

**A11.** いいえ。`match` は多くのうちの一つ(sum type)のためのもの。数値には `if` を使います。道具は2つあり、それぞれ役割が分かれています。用途が中途半端に重なって、覚えることを増やしたくはないのです。

---

**Q12.** ペイロードには何を入れられますか？

**A12.** スカラーと素朴な構造体 — `Wrap(Point)` は大丈夫で、アーム `Wrap(p) => p.x + p.y` は中身にアクセスできます。(`string` のような所有型のペイロードは、今は受け付けません。)

---

**Q13.** これをモデル化してください: 取得中のページは *読み込み中* か、*サイズ付きで準備完了* か、*コード付きで失敗* か。

**A13.**

```align
Page { Loading, Ready(i64), Failed(i64) }

fn describe(p: Page) -> i64 = match p {
    Loading   => 0,
    Ready(n)  => n,
    Failed(c) => -c,
}
```

ありえない状態 — 準備完了 *かつ* 失敗、ページのないサイズ — は書けません。それこそが sum type の *目的* です。

---

> **第六の戒律**
>
> *あるものが多くのうちの一つなら、その多くを言え。それから `match` し、リストの管理はコンパイラに任せよ。*

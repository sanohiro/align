# 5. Chains

> 🌐 [English](../05-chains.md) · **日本語**

**Q1.** これを左から右へ、声に出して読んでください:

```align
scores.map(fn s { s + 5 }).where(fn s { s >= 60 }).count()
```

**A1.** 「scores を取り、各点に5を足し、60以上のものを残し、それを数える。」読めるなら、書けます。それが chain の狙いです。

---

**Q2.** `scores := [55, 40, 90]` のとき、これは何を答えますか？

**A2.** `2`。全員に5点ずつ下駄を履かせた結果、`55` は `60` に（合格）、`40` は `45` に（不合格）なります。`90` は最初から疑いようがありません。

---

**Q3.** ステージの順序は問題になりますか？ 比べてください:

```align
xs.where(fn x { x > 0 }).map(fn x { x * 2 }).sum()
xs.map(fn x { x * 2 }).where(fn x { x > 0 }).sum()
```

**A3.** 今回は答えが一致します — 2倍しても符号は変わりませんから。でも *仕事量* が違います。前者は先に絞り込んでいるので、2倍にする回数が少なくて済みます。できるだけ早めに絞り込みましょう。chain は、あなたが書いたことを、書いた順に忠実に実行します。

---

**Q4.** ではこの2つは？

```align
xs.map(fn x { x - 10 }).where(fn x { x > 0 }).count()
xs.where(fn x { x > 0 }).map(fn x { x - 10 }).count()
```

**A4.** 答えが違います! `xs := [5, 15]` のとき、前者は引いてから正の数を残す — count は `1`。後者は正の数を残してから引く — count は `2`。chain は文であり、語順が意味を持つのです。

---

**Q5.** なぜ C や Go のように `for` ループを書かないのですか？

**A5.** `for` ループは「CPUの足をどう動かすか（How）」を指示する手段です。一方、パイプラインは「データが何に変わるか（What）」を宣言する手段です。あなたが「何を（What）」するかを宣言すれば、コンパイラは手書きのループよりもはるかに賢く「どうやって（How）」最適化するか（ベクトル化、融合、アンロール）を決定できるのです。

---

**Q6.** 他の言語では、`map` や `where` をチェーンさせると、ステップごとに一時的な配列がメモリ上に作られ、メモリを食いつぶします。Align もそうですか？

**A6.** いいえ。Align のパイプラインは、最後の「畳み込み（collapse）」が来るまで「遅延（lazy）」されます。コンパイラは `map`–`where`–`map`–`scan`–`sum` という5つのステージをすべて「たった1つのループ」に融合（fuse）し、中間結果はレジスタに留まります。まるであなたが苦労してC言語で手書きの最適化ループを書いたかのように振る舞いますが、あなた自身はそれを書く必要がないのです。確かめたければ: `alignc emit-llvm yourfile.align`。

---

**Q7.** では、長い chain を複数行に分けて書いたら、何かコストがかかりますか？

```align
total := items
    .where(.active)
    .price
    .map(with_tax)
    .sum()
```

**A7.** 何も。`.` で始まる行は chain を続けます。レイアウトは読み手のためのもの。コンパイラには、どちらも同じ1つの pipeline に見えています。

---

**Q8.** chain を途中で止めて、やりかけの仕事を持っておいてもいいですか？

```align
halfway := items.where(.active).price
```

**A8.** いいえ — chain は畳み込み(`sum`、`count`、…)か実体化(`to_array`、`sort`、`map_into`)で終わらなければなりません。途中の状態を保持するということは、未完成のループを隠し持つようなものです。終わらせるか、始めないかです。

---

**Q9.** では、絞り込んだ集合を2つの計算で *再利用* するにはどうしますか？

**A9.** 一度実体化して、二度問う:

```align
active := items.where(.active).price.to_array()
print(active.sum())
print(active.max())
```

目に見える確保が1回、低コストな畳み込みが2回。(いつか *グループ* ごとに *たくさんの* 集計が欲しくなります — それが10章の `agg` です。)

---

**Q10.** ここで `chunks` は何をしますか？

```align
[1, 2, 3, 4, 5].chunks(2).map(fn c { c.sum() }).to_array()
```

**A10.** `[3, 7, 5]` — `[1,2]`、`[3,4]`、`[5]` の和。`chunks(n)` は配列を `n` 個ずつの塊に分けます（最後の塊は端数になります）。それぞれが slice であり、ほかと同じようにパイプライン処理されます。

---

**Q11.** *あなた* が所有するメモリに書き込む chain — `map_into` とは何ですか？

```align
src.map(dbl).map_into(dst)
```

**A11.** メモリ確保を行わない終端処理です。結果は slice `dst` に入ります。`dst` は同じ長さでなければならず、`src` と重ならないことをコンパイラが証明します。バッファを再利用するホットパス向けです。

---

**Q12.** さて、習慣です。あなたはこれからループを書こうとしています。まず何を問いますか？

**A12.** *「これはどんな変換なのか?」* そうしたら、それをステージとして書きます: 各要素を変える → いくつか残す → 畳み込む。ステージに名前をつけられないなら、11章(`loop`)が待っています — でも、毎回まずはその問いから始めましょう。

---

> **第五の戒律**
>
> *chain は左から右へ読み、早めに絞り、そして終わる。ひとつの文、ひとつのループ、ひとつの答え。*

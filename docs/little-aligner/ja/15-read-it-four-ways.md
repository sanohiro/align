# 15. Read It Four Ways

> 🌐 [English](../15-read-it-four-ways.md) · **日本語**

**Q1.** 小さな呪文をたくさん覚えました。一つの小さなプログラムの中に、言語全体を見ることはできますか？

**A1.** 読み方を学ぶには、これで十分です。

```align
Record { score: i64, valid: bool }

fn total(data: str) -> Result<i64, Error> {
    arena {
        rows: soa<Record> := json.decode(data)?
        return Ok(rows.where(.valid).score.sum())
    }
}
```

一度読んでください。急いではいけません。

---

**Q2.** decode された3つの record が `(10, true)`、`(20, false)`、`(30, true)` だとします。`total` の答えは？

**A2.** `Ok(40)`。valid な行だけを残し、その score を取り出して足します。これが**答えの読み方**です。プログラムがどう動くかを問う前に、それが何を意味するのかを言います。

---

**Q3.** なぜ答えから始めるのですか？ 面白いのは仕組みのほうでは？

**A3.** `40` を変えてしまう最適化はバグだからです。間違った答えに美しい lifetime を与えても、やはり間違いです。意味が先にあります。ほかの読み方は、その意味を守らなければなりません。

---

**Q4.** 二度目は、データの通る道を読んでください。

**A4.** text が入る。`json.decode` がそれを column に変える。`where(.valid)` が位置を選ぶ。`.score` が一つの column を射影する。`sum()` が多数の score を一つの数へ畳み込む。これが**流れの読み方**です。左から右へ、それぞれの変換に名前をつけます。

---

**Q5.** `where(.valid).score.sum()` の中には、いくつの loop が隠れていますか？

**A5.** fusion された一つの loop です。ただし、関数全体が1パスだと言ってはいけません。text の decode はそれ自体が仕事であり、そのあとに column を走査する fusion 済みのパスがあります。実在する仕事を、多くも少なくもなく数えてください。

---

**Q6.** `Record` に `name`、`email`、`notes` もあったとします。fusion されたパスが読む column は？

**A6.** `valid` と `score` です。ほかの column も存在しますが、この問いは触れません。第9章の横倒しのレイアウトと、第5章の fusion が、いま一つの物語になりました。

---

**Q7.** 三度目は、それぞれを誰が所有し、どこで死ぬかを読んでください。

**A7.**

- `data` は呼び出し元から借りた view、`str` です。
- `rows` とその column は arena に住み、閉じる `}` で死にます。
- 最後の `i64` は Copy であり、`Ok` の中に入って arena の外へ出ます。

これが**寿命の読み方**です。値を見つけるだけではいけません。その住み家と、最後の行を見つけてください。

---

**Q8.** 数は arena の外へ出られるのに、なぜ `rows` は出られないのですか？

**A8.** 数は自分自身の bit を持ち、arena の何も指していません。`rows` の背後にあるのは arena の memory です。これを返せば、取り壊された建物を眺める view が残ります。type checker は、そんな物語が起きる前に拒みます。

---

**Q9.** JSON が壊れていたら？

**A9.** `?` が `total` から `Err` を返します。外へ出る途中でも arena は終わり、その memory は回収されます。error の流れと lifetime は一致します。早く出る扉が、memory leak になることはありません。

---

**Q10.** 四度目は、machine がどこで仕事をするかを読んでください。

**A10.** 入力を parse し、arena の pointer を進めて column を確保し、それから必要な2つの column を fusion された一つの loop で走査します。filter 済みの array は作らず、string の clone もしません。これが**仕事量の読み方**です。パス、触れる column、allocation、copy、そして出口を数えます。

---

**Q11.** filter 済みの array が作られていないと、どうしてわかるのですか？

**A11.** `where` と `sum` の間に、materialize する終端がないからです。array が欲しければ `.to_array()` と言わなければなりません。deep copy が欲しければ `.clone()` と言わなければなりません。Align では、仕事量の物語を変える操作に、発音できる名前があります。

---

**Q12.** 代わりに、こう書くと？

```align
rows.where(.valid).score.to_array().sum()
```

**A12.** 答えは同じです。仕事量は違います。`to_array()` は score を目に見える形で materialize し、`sum()` はその array をもう一度走査します。array が必要なこともありますが、ここでは必要ありません。答えの読み方が一致する一方、仕事量の読み方が違いを見つけます。

---

**Q13.** `?` を `else` に置き換えられますか？

**A13.** 同じ型の fallback 値を選べば可能です。それは答えの読み方を変えます。壊れた入力は failure として上へ渡らず、意図的に別の値へ置き換えられます。小さく見える syntax が、大きな方針を運ぶことがあります。punctuation だけでなく、その方針を読んでください。

---

**Q14.** このプログラムのどこに `loop` が入りますか？

**A14.** column の走査には入りません。行はすでに既知の collection であり、多数の要素に一つの操作をします — これは pipeline です。`loop` が属するのは、制御が円を描く境界です。EOF まで次の block を読む、ready になるまで retry する、安定するまで収束させる、といった場所です。

---

**Q15.** では `match` が入るのは？

**A15.** データが「多くのうちの一つ」である場所です。decode error の詳細を調べるときや、複数の variant を持つ sum type の field を処理するときです。`match` は値の形に従い、`loop` は制御の形に従い、pipeline は多数の値の流れに従います。

---

**Q16.** では、4つの読み方。前を見ずに言ってください。

**A16.**

1. **答え** — これはどんな値、あるいは failure を意味するか？
2. **流れ** — データはどう変わり、制御はどこへ行くか？
3. **寿命** — それぞれの値を誰が所有し、どこで死ぬか？
4. **仕事量** — machine はどの byte、パス、allocation、copy、出口を実行するか？

一つでも答えが霞んでいるなら、プログラムのうち、まだ理解していない場所を見つけたということです。

---

**Q17.** なぜ Align では、この4つの読み方が同じ source に収まるのですか？

**A17.** Align の変わった規則は、もともと別々の小技ではなかったからです。pipeline は流れを明白にし、仕事を fusion できるようにします。`soa` は触れる column を明白にします。Move と arena は寿命を明白にします。`?`、`.clone()`、`.to_array()` は出口、copy、allocation を明白にします。この言語は、人間、compiler、hardware に、一つの正直な物語を読ませます。

---

**Q18.** 本を閉じて答えてください。要求が「100より大きい valid な score の数を返す」に変わりました。最後の式だけを変えると？

**A18.**

```align
return Ok(rows
    .where(.valid)
    .score
    .where(fn s { s > 100 })
    .count())
```

前の chain を写さず、文からこれを書けたなら、pipeline の章は道具になっています。

---

**Q19.** 新しい要求は「100より大きい valid な score をすべて返す」。何が変わる？

**A19.** 終端が `.to_array()` になり、関数の return type と lifetime の物語も変わります。要求の「数」と「すべて」という一語の違いが、scalar answer を owned collection に変えます。

---

**Q20.** 新しい要求は「malformed row の理由を返し、なければ count」。変えてはいけない古い選択は？

**A20.** `Result` と `?` を保ちます。failure には caller が求めた物語があります。`else` fallback は、いま求められている答えそのものを消してしまいます。

---

**Q21.** 新しい要求は「EOF まで block を処理し、各 block 内で valid score を合計する」。現れる2つの control shape は？

**A21.** block の外側に `loop`、各 block の内側に pipeline。一つの問題が複数の形を持つことがあります。習熟とは、それぞれの境界に一つの適した道具を与えることです。

---

**Q22.** 新しい要求は「input は巨大で count 一つだけ」。`soa` か scanner か？

**A22.** scanner。あとで再利用しない column を保存してはいけません。答えの読み方が小さいままなら、仕事量と lifetime の読み方も小さくします。

---

**Q23.** 新しい要求は「20個の分析が同じ5 column を再利用する」。今度は？

**A23.** arena 内の `soa`。一度 decode し、phase のあいだ batch を保ち、分析に column を流します。workload が本当に違うから、同じ言語が異なる答えを持ちます。

---

**Q24.** Align の機能を学んだ、と言えるのはいつ？

**A24.** 定義を言えるときではありません。新しい問題に対して、それを選び、退け、古い道具と組み合わせ、実行前に答え・寿命・cost を言えるときです。

---

> **第十五の戒律**
>
> *一度目は答えを、二度目は流れを、三度目は寿命を、四度目は仕事量を読め。4つの物語が一致したとき、あなたはそのプログラムを理解している。*

これでドリルは終わりです。もう、自分の問題を Align で解き始めるだけのものを持っています。data の形を決め、flow を選び、lifetime を述べ、仕事量を数えられます。[ガイド](../../guide/ja/README.md)は、実際の program が必要としたときに、文字列、module、標準ライブラリ、system の組み立て方という広い surface に名前を与えます。それは次の道具が並ぶ棚であり、aligner になり始める場所ではありません。それは、この本の中で起きました。

*さあ、何か作りに行きましょう。*

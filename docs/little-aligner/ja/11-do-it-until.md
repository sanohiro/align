# 11. Do It Until

> 🌐 [English](../11-do-it-until.md) · **日本語**

**Q1.** Align の `for` ループはどこにありますか？

**A1.** ありません。`while` もありません。あなたは 10 章のあいだ、それを気にせずやってきました — パイプラインこそがループだったのです。

---

**Q2.** パイプラインがそんなに完璧なら、なぜわざわざ `loop` を用意したのですか？

**A2.** パイプラインはデータが「まっすぐ」流れるとき — 既知のスタート地点からゴールへと向かうとき — のためのものです。しかし、グラフの経路探索や、ソケットからの応答を待つ処理など、一部の問題は「円」の形をしています。`loop` は、アルゴリズムそのものが円を描くときのために存在します。しかし注目してください: Align では `loop` すらも値を返す「式」です。これもまた、自ら靴ひもを結ぶような、ひとつの特殊なパイプラインステージにすぎないのです。

---

**Q3.** その円を見せてください。ファイルを 4096 バイトずつ、最後まで（空になるまで）読むには？

**A3.** この言語が持つ、たったひとつのループ構文の出番です。

```align
fn pump(r: reader, w: writer, buf: buffer) -> Result<(), Error> {
    loop {
        n := r.read(buf)?
        if n == 0 { break Ok(()) }
        w.write(buf.bytes())?
    }
}
```

`loop` は、`break` が走るまでブロックを繰り返します。空になるまで繰り返し、そして外へ出ます。

---

**Q4.** そこの `break Ok(())` は何をしましたか？

**A4.** 一度に 2 つのことをしました。ループを終わらせ、そして値を手渡しました。`loop` は**式**です — `if` のように、`match` のように。ループは関数の最後の式なので、ループの値がそのまま関数の答えになります。

---

**Q5.** では、reader が渡してくるものをすべて合計するには？

**A5.**

```align
mut total := 0
sum := loop {
    n := r.read(buf)?
    if n == 0 { break total }
    total = total + n
}
```

途中までの答えは、ループの*前*に宣言した `mut` のローカル変数に保持しておきます。`break total` が、周回を終えたときにその値をループの外へと運び出します。

---

**Q6.** Q3 のように、`？` と `loop` はひとつの関数で同居できますか?

**A6.** たったいま同居していました。`?` は**関数**から抜けます — どんな失敗も、ただちに `Err` となって関数全体から抜け出します。`break` は**ループ**から抜けます。2 つの扉、はっきりと札が掛かっていて、決して同じ扉ではありません。

---

**Q7.** `continue` はどこですか？

**A7.** ありません。次の周回へスキップしたいなら、本体の残りを `if` で囲みます。2 つのループを一度に抜けたいなら — その内側のループは別の関数に切り出すべきです。出口はひとつ、`break` です。

---

**Q8.** あなたは `loop` の中で `i` を `0` から `len` まで数え、毎周 `xs[i]` に同じことをしています。何をしてしまいましたか？

**A8.** 変装した `for` ループを書いてしまいました。変装を解きましょう — それは最初から `xs.map(...)` だったのです。パイプラインは「データ」の処理を受け持ち、`loop` は「制御フロー」を受け持つからです。コンパイラもそう言います(lint が付きます)。

---

**Q9.** では再帰は？ 昔の本は、自分自身を呼ぶ関数こそがループだと言っています。

**A9.** ここでは違います。再帰による「ループ」は、周回ごとにスタックフレームを 1 つ使います — Align は末尾呼び出しの魔法を約束しませんし、そもそもスコープ末尾の drop と `?` が、そうした魔法をこっそり壊してしまいます。1000回の繰り返し処理のために、1000個のスタックフレームを消費するべきではありません。`loop` はそのためにあります。

---

**Q10.** では、再帰は何のためにありますか？

**A10.** 再帰的な「構造」を持った問題 — 入れ子になるもの — のためです。パーサの中の括弧の中のパーサ。子ノードを持つ木構造などです。**データ**が再帰するとき、関数もそうしてよいのです。再帰はデータの「形」を表現するためのものであり、繰り返し「回数」のためのものではありません。

---

**Q11.** いつパイプラインで、いつ `loop` で、いつ再帰するのか。最後の組分け帽子です。

**A11.**

- 多くの要素に同じ操作 → **パイプライン**(2〜5 章)。
- グループごとの畳み込み → **`group_by`**(10 章)。
- 何周するか、始めてみるまでわからない（空になるまで少しずつ読む、成功するまでリトライする、収束計算など） → **`loop`** を使い、`break value` で結果を返します。
- データそのものが入れ子になる → **再帰**、そのかたちのために。

---

> **第十一の戒律**
>
> *複数の要素にはパイプラインを、条件を満たすまでには `loop` を、入れ子には再帰を。そして、そもそもループする前に — それがパイプラインではなかったか、と問え。*

---

**Q12.** これで終わりですか？

**A12.** ドリルは、はい。Align は — [ガイド](../../guide/ja/README.md)に残りがあります: 文字列、JSON、並列処理、SIMD、標準ライブラリ。でもあなたはもう、Align を外国語のようには読みません。知っている道具のカタログのように読めるはずです。

*さあ、何か作りに行きましょう。*

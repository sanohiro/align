# 8. Whose Is It?

> 🌐 [English](../08-whose-is-it.md) · **日本語**

**Q1.**

```align
a := 42
b := a
```

いま 42 はいくつ存在しますか?

**A1.** 2つ。数は **Copy** です。代入によって複製されます。それぞれ独立した値になります。

---

**Q2.**

```align
a := "hello".clone()    // an owned string
b := a
```

いま所有された string はいくつ存在しますか?

**A2.** 1つ。`string` はヒープ上のバッファを所有し、所有型の代入は **Move** になります。バッファが持ち主を替えたのです。今は `b` がそれを持っています。

---

**Q3.** では、その直後の `print(a.len())` は何をしますか？

**A3.** コンパイルできません: *ムーブ済みの値 `a` の使用*。古い変数はもはや使えません。1つのバッファに対して所有者は常に1人であり、例外はありません。

---

**Q4.** なぜそんなに厳しいのですか？

**A4.** 所有者が1人なら、誰がいつメモリを解放すべきかコンパイラが正確に把握できるからです。ガベージコレクタを待つ必要も、二重解放に怯える必要も、ライフタイム注釈を書く必要もありません。この厳しさこそが、メモリ管理そのものなのです。

---

**Q5.** でも、本当に string が2つ欲しいのです。

**A5.** それなら、相応のコストを払ってください: `b := a.clone()`。コストがかかる場所では、明示的にディープコピーを書いてください。Align では何でもコピーできます — ただ、*見えないところで* コピーすることだけができないのです。

---

**Q6.** どの型が Move ですか？

**A6.** 所有者たち: `string`、`array<T>`、`buffer`、`box`、ファイルハンドルなどです。単純なものはすべて — 数、`bool`、view、それらの小さな構造体 — Copy です。リソースを所有するなら Move。ただのデータなら Copy。そして `string` を *含む* 構造体もまた Move になります — 所有権は上位の型へと伝播するからです。

---

**Q7.** どんな `.clone()` もする前の `"hello"` は何ですか — このリテラルは誰が所有していますか？

**A7.** 誰も。これは `str` — **view** です。ポインタと長さだけで、自分より長生きするバイト列を指し示しているだけです。view は Copy です — データへの「ビュー（見え方）」をコピーするのはタダだからです。

---

**Q8.** さて arena です。これは何を表示しますか？

```align
fn shout(name: str) -> string {
    arena {
        s := template "hey, {name}!"
        return s.clone()
    }
}

print(shout("align"))
```

**A8.** `hey, align!`。`arena { }` の中では、template は arena 内にメモリを確保します。`}` で、arena 全体が一括で解放されます。`.clone()` が、その前に生き残るべきデータを外へコピーしてくれたのです。文字列の `+` はコンパイルエラーであり、文字列の連結には builder しか使えません。

---

**Q9.** clone なしで `return s` としたら？

**A9.** コンパイルできません: *arena で確保した値は返せません*。コンパイラは `s` のリージョンと、それが `}` で破棄されることを知っています。外へ持ち出すにはコピーが必要であり、それは目に見える形で明示しなければなりません: `.clone()`。

---

**Q10.** arena に手を伸ばすのはいつですか？

**A10.** ある *フェーズ* が、同時に解放される多くのものを確保するとき — ファイルのパース、リクエストの処理、バッチのデコードなどです。フェーズを `arena {}` でひとくくりにすれば、一時領域の確保はポインタを1つ進めるだけのコストで済みます。後始末はスコープを抜けるだけでよく、解放忘れもありません。

---

**Q11.** 他の言語では、ガベージコレクタが私の落としたゴミを見つけて掃除してくれます。なぜ代わりにそれを使わないのですか？

**A11.** ガベージコレクタは、あなたのプログラムの後ろをついて回り、一つひとつのゴミを拾い集める清掃員です。一方、`arena` は「建物」です。仕事が終わったら、建物ごと爆破して取り壊します。個別のミスを拾い集めるのではなく、最初から「寿命」を丸ごと計画しておくための仕組みなのです。

---

**Q12.** もしループの中で10,000個の一時的な文字列を作り、それを1つずつ解放したとしたら、コストはどうなりますか？

**A12.** 他の言語なら、汎用アロケータを10,000回往復し、その一回一回にロックと管理台帳の更新が付いてきます。しかし `arena` の中なら、10,000個の文字列を作るコストは、ただポインタを10,000回進めるだけです。arena とは単なる「後始末」の機能ではなく、*目が眩むほどの確保スピード（allocation speed）*を得るための機能なのです。

---

**Q13.** では、あらゆる新しいデータについての、決定まるごとは？

**A13.** 問いは一つ — *それはどれだけ生きるのか?*

- このスコープ → 単純な値、それで十分
- このフェーズ → arena、生き残るものは `.clone()`
- もっと長く、所有者1人 → 所有型、move して持ち回る
- ただ眺めているだけ → view、タダ

---

**Q14.** では、いったい誰のものですか — この行の、まさにこのバッファは？

**A14.** ソースを読んでください: 最後に move された先の変数です。Align の所有権は実行時の謎ではありません。コードに明記されており、あなたの解釈が正しいかはコンパイラがすでに検証してくれています。

---

> **第八の戒律**
>
> *所有者は一度に1人。短命なものは arena の中で管理せよ。そしてどうしても2つ要るなら — `.clone()`、皆に見える場所で。*

# 9. Turn It Sideways

> 🌐 [English](../09-turn-it-sideways.md) · **日本語**

**Q1.** テーブルに3人のユーザー、1行に1人:

```text
alice   30   true
bob     25   false
carol   41   true
```

すべての **age** を読んでください。あなたの目はどちらに動きますか?

**A1.** 列を下へ — 各行の名前とフラグを読み飛ばしながら。

---

**Q2.** では、テーブルが *横倒し* に書かれていたら — 1行に1フィールド？

```text
name:   alice  bob  carol
age:    30     25   41
active: true   false true
```

**A2.** まっすぐ横へ1行だけを読みます。ほかのものには一切触れません。メモリの好みは、あなたの目の好みと同じです。行優先が **AoS**(array of structs)、横倒しが **SoA**(struct of arrays)です。

---

**Q3.** なぜ機械はそれを気にするのですか？

**A3.** キャッシュラインです。メモリは64バイトずつ、まとめてフェッチされます。AoS では、age を1つ読み込むたびに、頼んでもいない名前とフラグがついてくる。SoA では、age を読み込めば一度に16個の age が手に入ります。そして SIMD（ガイドの12章）は連続したデータしか処理できません — だからこそ、「列」が SIMD のレーンに最適な餌となるのです。

---

**Q4.** 横倒し（SoA）がそれほどまでに速いのなら、なぜプログラミングの世界はオブジェクト指向（AoS）を発明したのですか？

**A4.** 人間が「エンティティ（個体）」で思考するからです。私たちは「User」という概念があり、それが名前や年齢を持っていると想像します。しかし CPU にはエンティティなど見えません。ただ平坦に続くバイトの列が見えるだけです。オブジェクト指向は「人間の想像力」を優先しました。一方、データ指向設計（SoA）は「シリコンの物理的な現実」を優先するのです。

---

**Q5.** Align ではデータをどう横倒しにしますか？

**A5.** 呼び出し1つ、arena の中で:

```align
User { active: bool, score: i64, age: i64 }

arena {
    mut s := rows.to_soa()      // soa<User> — three columns now
    ...
}
```

依然として `soa<User>` です — あなたは `User` で考え続けます。変わったのはメモリ上のレイアウトだけです。

---

**Q6.** `s.age` は何ですか？

**A6.** age の **列** — `i64` の単純な slice です。そして slice ということは、ここまでの全章が当てはまります: `s.age.sum()`、`s.age.max()`、`s.age.map(...)...`。

---

**Q7.** 上の3人のユーザーで、`s.age.sum()` は何ですか？

**A7.** `96`。メモリ上に密に詰まった1行を、そのまま合計しました。

---

**Q8.** `s.where(.active).age.sum()` は何ですか？

**A8.** `71` — `30 + 41`。bob は非アクティブです。触れた列は2つ(絞るための `active`、足すための `age`)。name 列は RAM からキャッシュにすら読み込まれていません。これがこの仕組みのすべてであり、そして非常に強力なのです。

---

**Q9.** それでも alice を丸ごと — 行まるごと — 取り戻せますか？

**A9.** `u := s[0]` が、3つの列から `User` を1人分かき集めます。列単位の読み込みよりコストがかかります(1回ではなく3回のフェッチ)— 横倒しのレイアウトは、「列」でコストを節約する代わりに、「行」を復元するときにそのツケを払うことになるのです。

---

**Q10.** では1セルは？ 書き込みは?

**A10.** `s[0].age` は1セルを読みます。`mut` な soa なら `s[1].score = 99` は直接、列の中の1つのセルに書き込みます。

---

**Q11.** 特定の範囲(ウィンドウ)や、列の途中から読み込む場合は？

**A11.** `s.age[1..3].sum()` — 列を、ほかの slice と同じように切り出します: `25 + 41 = 66`。

---

**Q12.** 横倒しのレイアウトが *負ける* のはいつですか？

**A12.** 行を丸ごと、まれに、単発で触れるとき — 設定レコード1つ、リクエスト1つなどです。すべてのフィールドをかき集めるのは、SoA がバラバラに配置したものを再び組み立てる作業だからです。行単位で丸ごと扱うなら AoS。大量に走査する列なら SoA。

---

**Q13.** データは JSON で届きます。まず行を作って、あとで横倒しにしなければなりませんか？

**A13.** いいえ — *じかに* 横倒しに decode します:

```align
s: soa<User> := json.decode(data)?
```

パーサは読みながら列を埋めます。途中で「行」の形をした一時オブジェクトを作ることはなく、string の列は JSON テキストそのものへの view です。

---

**Q14.** なぜどの soa も `arena` の中で暮らしていたのですか？

**A14.** 8章の答えです: 列はライフタイムを一つ共有する *バッチ* です — 一度の decode から生まれ、分析が終わると同時に破棄されます。arena がそれを宣言し、コンパイラがあなたにそれを守らせます。

---

**Q15.** 習慣を言ってください。

**A15.** *フィールドを大量に走査するなら、入り口でデータを横倒しにする* — `to_soa()`、あるいは `json.decode` でじかに `soa<T>` へ — *そしてそこから先は「列」単位で処理する。*

---

> **第九の戒律**
>
> *行ではなく列を走査せよ。データは入り口で一度だけ横倒しにせよ。そうすればコンピュータが喜んで効率的に動いてくれる。*

# 9. Turn It Sideways

> 🌐 [English](../09-turn-it-sideways.md) · **日本語**

**Q1.** テーブルに3人のユーザー、1行に1人:

```text
alice   30   true
bob     25   false
carol   41   true
```

すべての **age** を読んでください。あなたの目はどちらに動きますか?

**A1.** 列を下へ — 行ごとに名前とフラグを飛び越えながら。

---

**Q2.** では、テーブルが *横倒し* に書かれていたら — 1行に1フィールド?

```text
name:   alice  bob  carol
age:    30     25   41
active: true   false true
```

**A2.** まっすぐ1行を横へ、ほかには何も触れずに。メモリの好みは、あなたの目の好みと同じです。行優先が **AoS**(array of structs)、横倒しが **SoA**(struct of arrays)です。

---

**Q3.** なぜ機械はそれを気にするのですか?

**A3.** キャッシュラインです。メモリは64バイトずつ、ひと飲みで届きます。AoS では、age をひと飲みするたびに、頼んでもいない名前とフラグがついてくる。SoA では、age をひと飲みすれば *16個の age* です。そして SIMD(ガイドの12章)は連続したレーンしか食べません — 列 *こそが* レーンへの給餌です。

---

**Q4.** Align ではデータをどう横倒しにしますか?

**A4.** 呼び出し1つ、arena の中で:

```align
User { active: bool, score: i64, age: i64 }

arena {
    mut s := rows.to_soa()      // soa<User> — three columns now
    ...
}
```

依然として `soa<User>` です — あなたは `User` で考え続けます。変わったのはメモリだけ。

---

**Q5.** `s.age` は何ですか?

**A5.** age の **列** — `i64` の素朴な slice です。そして slice ということは、ここまでの全章が当てはまります: `s.age.sum()`、`s.age.max()`、`s.age.map(...)...`。

---

**Q6.** 上の3人のユーザーで、`s.age.sum()` は何ですか?

**A6.** `96`。密に詰まったメモリの1行、それを合計しました。

---

**Q7.** `s.where(.active).age.sum()` は何ですか?

**A7.** `71` — `30 + 41`。bob は非アクティブです。触れた列は2つ(絞るための `active`、足すための `age`)。名前は RAM から一度も出ていません。それがこの手品のすべてであり、しかも大したものです。

---

**Q8.** それでも alice を丸ごと — 行まるごと — 取り戻せますか?

**A8.** `u := s[0]` が、3つの列から `User` を1人分かき集めます。列の読み込みより高くつきます(1回ではなく3回のフェッチ)— 横倒しのレイアウトは、*列* で節約した分を *行* で請求するのです。

---

**Q9.** では1セルは? 書き込みは?

**A9.** `s[0].age` は1セルを読みます。`mut` な soa なら `s[1].score = 99` が1つ書きます — まっすぐ列の中へ。

---

**Q10.** 窓は? 列の真ん中は?

**A10.** `s.age[1..3].sum()` — 列を、ほかの slice と同じように切り出します: `25 + 41 = 66`。

---

**Q11.** 横倒しのレイアウトが *負ける* のはいつですか?

**A11.** 行を丸ごと、まれに、単発で触れるとき — 設定レコード1つ、リクエスト1つ。全フィールドをかき集めるのは、SoA が集めたものを再びばらまくこと。丸ごと扱う行なら AoS。大量に走査する列なら SoA。

---

**Q12.** データは JSON で届きます。まず行を作って、あとで横倒しにしなければなりませんか?

**A12.** いいえ — *じかに* 横倒しに decode します:

```align
s: soa<User> := json.decode(data)?
```

パーサは読みながら列を埋めます。行の形をした仲介役はなく、string の列は JSON テキストそのものへの view です。

---

**Q13.** なぜどの soa も `arena` の中で暮らしていたのですか?

**A13.** 8章の答えです: 列はライフタイムを一つ共有する *バッチ* です — 一度の decode から生まれ、一度の分析のあとに死ぬ。arena がそれを宣言し、コンパイラがあなたにそれを守らせます。

---

**Q14.** 習慣を言ってください。

**A14.** *フィールドを大量に走査するなら、入り口でデータを横倒しにする* — `to_soa()`、あるいは `json.decode` でじかに `soa<T>` へ — *そしてそこから先は列で語る。*

---

> **第九の戒律**
>
> *行ではなく列を走査せよ。データは入り口で一度だけ横倒しにせよ。そうすれば機械が一日じゅう感謝する。*

# メモリ: 値、arena、heap

> 🌐 [English](../05-memory.md) · **日本語**

Align にはガベージコレクタがなく、手動の `free` もなく、ライフタイム注釈もありません。代わりにあるのは、**データがどこに住むかは自分で下す決定**であり、所有権は**型の性質**であり、コンパイラはあらゆるライフタイムを推論して、宙ぶらりんの参照(dangle)を生むプログラムを拒否する、という仕組みです。この章がモデルのすべてです —— そしてそれは小さい。

## 値(既定)

たいていのデータは素の値です。数値、`bool`、小さな構造体、スカラーのタプル。値はスタック上に住み、**Copy** です —— 代入や引き渡しをするとコピーされ、両方のコピーは独立していて、何も allocate されていないので何も解放されません。

```align
Point { x: f64, y: f64 }

fn main() -> i32 {
    p := Point { x: 1.0, y: 2.0 }
    q := p              // a copy; p and q are independent
    return 0
}                       // scope ends, values are simply gone
```

構造体が大きくなると、値渡しにはコストがかかり始めます —— キャッシュライン 2 本を超えると、コンパイラは警告します(`huge struct copy`)。それは、コピーの代わりにスライスや arena、あるいは SoA([11](11-data-oriented.md) 章)に手を伸ばす合図です。

## Move 型 —— 所有権を型に持たせる

heap リソースを所有する型 —— `string`、`array<T>`、`buffer`、`box`、I/O ハンドル —— は Copy ではなく **Move** です。代入すると所有権が移り、その時点で元の束縛は死にます。コンパイラがそれを強制します。

```align
fn main() -> i32 {
    a := "hi".clone()   // a `string` — an owned heap buffer
    b := a              // ownership moves to b
    print(a.len())      // error: use of moved value 'a'
    return 0
}
```

このコンパイルエラーこそが、ガベージコレクタと Rust の可視ライフタイムの両方を置き換えるメカニズムのすべてです。どの時点でも所有者はちょうど 1 人なので、コンパイラはいつ解放すべきかを正確に知っています —— 二重解放も、解放後使用も、注釈もありません。所有者がスコープを抜けるとバッファは drop されます。`mut` の所有者に再代入すると、古い値が先に drop されます(リークなし)。本当に 2 つ欲しいときは、そう言います。`.clone()` は、目に見えるディープコピーです。

所有権を持つフィールド(`name: string`)を含む構造体は、それ自体が Move 型になり、再帰的に drop されます —— 所有権は構造をたどって合成されます。そうしたフィールドを読むこと(`u.name.len()`)は、何も消費せずに `str` ビューとして借用します。

## arena —— ライフタイムでまとめて確保する

作業の*ある局面*が、いっしょに死ぬ多くのものを確保するとき —— ファイルをパースする、リクエストを組み立てる、バッチをデコードする —— その局面を `arena {}` で包みます。

```align
fn join(a: str, b: str) -> string {
    arena {
        c := template "{a}{b}" // arena-backed temporary
        return c.clone()    // copy the result out — visible escape
    }                       // everything else freed here, O(1)
}

fn main() -> i32 {
    s := join("fu", "sion")
    print(s)                // fusion
    return 0
}
```

arena はバンプアロケータです。確保はポインタのインクリメントで、解放は、どれだけ確保したかに関わらずブロック全体でひとつの操作です。オブジェクトごとの帳簿づけもなく、断片化もありません。これがデータ指向のメモリ管理です。確保をオブジェクト単位ではなく、ライフタイム単位でまとめます。

コンパイラは、あらゆる値の **region** を追跡します。arena 内で確保された値は、そこから外へ出られません。

```align
fn leak(a: str, b: str) -> str {
    arena {
        s := a + b
        return s        // error: cannot return a value allocated in an arena
    }                   //        (it is freed at block end)
}
```

region に注釈を付けることは決してありません。region が姿を見せるのは、宙ぶらりんの参照を書こうとしたときだけです —— そしてそのプログラムはコンパイルできません。抜け道は常に同じで、常に目に見えます。`.clone()` がデータを所有された値へとコピーして外に出します。

## heap を、明示的に

`heap.new(x)` は、単一の明示的な確保 —— `box` —— を、外側を囲む arena の中に作ります。`.get()` で値を読み戻します。

```align
fn main() -> i32 {
    arena {
        b := heap.new(42)
        print(b.get())      // 42
    }
    return 0
}
```

これを書くことはめったにありません —— そして書く必要がなかったときは、コンパイラが教えてくれます。上の例は、実は lint を招きます。*「不要な heap 確保: この box はエスケープしていません —— 値を直接使ってください。」* `heap.new` が存在するのは、ある値が、それを計算したスタックフレームより長く生き延びる必要があり、しかも選んだ arena の中で生きるべき場合のためです。これは意図的に、日常の道具ではありません。Align では「すべてのオブジェクトを heap に確保する」というのはスタイルではありません —— 言語はあなたを値と arena へと導き、脱線すればそう言ってくれます。

## ビュー: `str` と `slice<T>`

`str` は文字列データの借用ビューであり、`slice<T>` は配列の借用ビューです。ビューは安価な Copy 値(ポインタと長さ)で、**それが指すデータの region** を帯びています —— arena データのビューは arena を出られず、構造体フィールドのビューはその構造体より長生きできません。同じ推論、同じ規則、注釈なし。

```align
fn main() -> i32 {
    xs := [10, 20, 30, 40]
    s := xs[1..3]           // slice view: elements 1 and 2, no copy
    print(s.sum())          // 50
    return 0
}
```

## 判断の手順

これで全部です。データを作るときは、ひとつだけ問うてください —— *そのライフタイムは何か?*

- この式かこのスコープで死ぬ → **値**。何もしません。
- ある局面の終わりにいっしょに死ぬバッチ → **arena**。生き残るものは `.clone()` で外に出します。
- フレームより長生きすべきひとつの値 → そのライフタイムに合う arena の中で **`heap.new`**。
- 他人のデータをただ眺めるだけ → **ビュー**(`str`、`slice`)。無料です。

それ以外のすべて —— いつ解放するか、エスケープするか、誰が何を所有するか —— はコンパイラの仕事です。コンパイル時に検査され、ソース上では見えません。ただし目に見える 2 点を除いて。ライフタイムが始まり終わる `arena {}` と、コピーの対価を払う `.clone()` です。

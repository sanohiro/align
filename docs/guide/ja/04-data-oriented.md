# データ指向: structure-of-arrays

> 🌐 [English](../04-data-oriented.md) · **日本語**

この章こそが Align の存在理由です。メモリ上でデータをどうレイアウトするかが、それをどれだけ速く処理できるかを — しばしば桁違いに — 決めます。そして Align は、その高速なレイアウトをファーストクラスの機能として提供します。

## Array-of-structs(構造体の配列)vs structure-of-arrays(配列の構造体)

100万個の粒子(パーティクル)があり、それぞれが位置(position)と速度(velocity)を持つとします。素直なレイアウトは array of structs(AoS、構造体の配列)です。

```text
[ {x0,y0,vx0,vy0}, {x1,y1,vx1,vy1}, ... ]
```

さて、すべての位置をその速度分だけ進めたいとします — `x`、`y`、`vx`、`vy` に触れることになりますが、もし本当に必要なのが `x` だけだったとしても、構造体全体をキャッシュに引きずり込むことになります。さらに悪いことに、全粒子にわたって `x` フィールドだけを更新しようとすると、欲しい値は数ワードごとに散らばっており、SIMD はそれらを連続してロードできません。

structure-of-arrays(SoA、配列の構造体)レイアウトは、各フィールドをそれぞれ専用の連続した配列に格納します。

```text
x:  [x0, x1, x2, ...]
y:  [y0, y1, y2, ...]
vx: [vx0, vx1, vx2, ...]
vy: [vy0, vy1, vy2, ...]
```

これなら「すべての速度をすべての位置に加える」処理は、密な2本の配列を歩調を揃えて(lockstep で)走査するだけになります — 完璧なキャッシュ挙動であり、ベクトライザは1命令あたり8レーンや16レーンをロードできます。これは GPU や ML エンジンが要求するレイアウトであり、信号処理が望むレイアウト(インターリーブされた `RGB RGB` ではなく、プレーナーな `RRR GGG BBB`)でもあります。

## Align は SoA を「作業」ではなく「宣言」にする

ほとんどの言語では、AoS を SoA に変換するのは手作業であり、間違いの起きやすい書き換えです。Align では、`soa<T>` が構造体定義から SoA レイアウトを与えてくれます — 形(shape)を一度宣言すれば、コンパイラがカラム(列)のレイアウトを行います。あなたは相変わらず `T` という単位で考えればよく、その裏でメモリは列指向(columnar)になっています。

## 列指向の reduction と group_by

データがすでに列指向になっているため、集約操作は自然かつ高速です。`group_by` はキーによって行をパーティション分割し、各グループを reduce します — 分析の中心にあるデータベース的な操作です — そしてそれは分岐のない(branchless)内側ループで、カラムに対して実行されます。

## 身につけるべき習慣

データがまとめて処理される場合 — 特に、繰り返し走査される数値データやレコードデータの場合 — SoA に手を伸ばしてください。目安としては、コレクションをループする際に一度に1つか2つのフィールドしか触らないなら、AoS はあなたの足を引っ張っており、SoA が味方になります。Align は、その選択に対して、普通のパイプラインコードから、キャッシュ密度と自動ベクトル化という報酬を無償で与えてくれます。

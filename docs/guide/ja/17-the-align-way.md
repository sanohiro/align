# Align の流儀

> 🌐 [English](../17-the-align-way.md) · **日本語**

Align におけるプログラミングのイディオム（定石）を集めました。これらはすべて、前章までのどこかで解説されたものです。ここでは、熟練の Align プログラマが無意識のうちに（自動操縦で）実行しているチェックリストとしてまとめます。

## 反復するな、変換を記述せよ

「ループ処理を書こう」と思い立つよりも先に、`map`、`where`、`reduce` などのパイプライン操作に手を伸ばしてください。そもそも Align には `for` や `while` は存在しません。パイプラインはコンパイラによって、高度にベクトル化された1パスのループへと自動的に融合されます。「EOF までデータを読み続ける」「ネットワーク要求をリトライする」「数値計算を収束させる」といった、真に逐次的な制御フローが必要な場合にのみ、値を返す `break` を伴う `loop` 式を使用します。大量のキー集約には `group_by` を使います。また、再帰関数はパーサや木構造の探索といった「本質的に再帰的な問題」のためのものであり、単なる反復処理の代用ではありません。もし `loop` ブロックの中でインデックスの数値を手動でインクリメントしているなら、それは明らかにパイプラインで書ける処理を見落としています。

```align
total := xs.map(f).where(p).sum()
```

## エラーは分岐ではなく `?` で扱え

失敗する可能性のある関数は `Result` を返し、呼び出し側は `?` 演算子でそれを受け取ります。エラー型を別の文脈へ伝播させる際は `map_err` を使って明示的に変換し、最終的にエラーを処理（消費）する地点で `match` 式を使って分岐させます。プログラムの終了コード（exit code）への配管作業は、`main() -> Result<(), Error>` に任せてください。

また、「値が存在しないこと」は `Option` と `else` で表現し、これは「操作が失敗したこと（`Result`）」とは明確に区別されます。どちらの状況を意図しているかは、関数のシグネチャが雄弁に語ってくれます。

## 寿命を選び、あとはメモリを忘れよ

- **スタック上のローカル値** → 何もする必要はありません。
- **1つの処理フェーズにおける一時的な割り当て** → `arena {}` を使います。ブロックを越えて生き残らせたいデータは `.clone()` でアリーナの外へ出します。
- **動的なテキストの組み立て** → `builder` を使います。ループの内部で文字列結合演算子 `+` は使いません。
- **他人が所有するデータ** → ビュー（`str` や `slice`）として借用します。メモリコストは無料です。

作業のフェーズごとに、ライフタイムの決定を1つだけ下します。残りのルール（値のムーブ、スコープからのエスケープ、ドロップのタイミング）はすべてコンパイラが強制します。だからこそ、Align では「コンパイルが通りさえすれば、メモリ管理は正しい」という確信が持てるのです。

## 大量データは SoA でレイアウトせよ

プログラムのホットパス（頻繁に実行される箇所）が、「非常に多数の行（レコード）のうちの1〜2個のフィールド」にしかアクセスしていないのであれば、データレイアウトを転置（SoA 化）してください。データがプログラムに入ってくる地点で `to_soa()` を呼ぶか、JSON を直接 `soa<T>` へデコードします。文字列キーに対して集約を繰り返す場合 → 一度だけ `dict_encode` で整数化します。従来の AoS（構造体の配列）レイアウトは、「個別のレコードを丸ごと扱うが、大量の要素をループで回すことはめったにない」データのために残しておきます。

## 並列化は 2 語

`par_map` を採用するのは、要素ごとに適用する関数の計算コストが十分に高いときだけです（必ず計測してください。逐次パイプラインは自動的にベクトル化されるため、多くの場合 `par_map` よりも高速です）。`task_group` の `spawn` と `wait()?` は、性質の異なる非同期ジョブをまとめるためのものです。関数の「純粋性」はコンパイラによって推論されるため、並列処理のコンパイルが通ればデータ競合は存在しません。もしコンパイルエラーになったなら、コンパイラが「あなたのコードに潜む隠れた副作用」を見つけ出してくれたということです。

## コンパイラに形を見せよ

Align のパフォーマンスの源泉は、「コンパイラがコードの性質を数学的に**証明**できること」にあります。連続したメモリ配置、エイリアシング（ポインタの競合）がないこと、Null 参照の不在、コールド（滅多に通らない）エラーパスの隔離、そしてアリーナによって縛られた明確なライフタイムなどです。言語に課されたあらゆる制限（Null の排除、一元化されたエラーモデル、Move セマンティクス、自動推論されるリージョン、終端が必須なパイプライン）は、コンパイラによるこれらの証明を、プログラマが注釈を書かなくても成立させ続けるために存在しています。

言語の「木目（設計思想）」に沿ってコードを書くこと（手動のインデックス操作よりもパイプラインを選び、番兵値（Sentinel value）よりも `Result` を選び、散らばった所有権よりもアリーナを選ぶこと）が、コンパイラの最適化エンジンに良質な情報を与え続けます。**「イディオムに沿って書かれた Align コード」は「高速に動作する Align コード」であり、この2つは設計上、完全に同義なのです。**

## 信じよ、しかしツールで検証せよ

日常の編集ループでは `alignc check` を回し、バージョン管理にコミットする前には `alignc fmt --write` で整形します。リンターが警告を発するとき（巨大な構造体のコピー、不要なヒープ割り当て、データが欠落しうるキャストなど）、それはパフォーマンスモデルが「ボトルネックになりうる特定の1行」を指し示しているということであり、その修正方法はたいていこのリストにあるイディオムのいずれかです。

そして、書いたパイプラインが「本当にベクトル化されたかどうか」が気になるときは、`alignc emit-llvm` で LLVM IR をダンプして確認してください。実際の出力結果（ダンプ）で確認できる事柄について、推測で議論してはいけません。

## 何も隠さない — キーワードでコードを読め

Align において、実行コストがかかる操作や危険を伴う操作はすべて、`grep` で検索可能なキーワードとしてコード上に自らを名乗ります。
- **割り当て**： `arena`、`heap.new`、`builder`、`.clone()`、`.to_array()`
- **失敗**： `Result`、`?`
- **ミューテーション**： `mut`、`out`
- **スレッド・並列処理**： `par_map`、`spawn`
- **チェックされない世界（FFIなど）**： `unsafe`、`raw`、`extern`

コードの読者は（それが人間であれ AI であれ）、これらのキーワードをスキャンするだけでファイルを監査できます。そして、これらのキーワードが存在しないことは、単なる「希望」ではなく、コンパイラに裏打ちされた「保証」なのです。ぜひ、この性質を保つような素直なコードを書いてください。あなたのプログラムを次に読む人（またはプログラム）こそが、この言語が設計された最大の理由だからです。

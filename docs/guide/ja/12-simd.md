# 明示的な SIMD ― vecN、マスク、アライメント

> 🌐 [English](../12-simd.md) · **日本語**

パフォーマンス系の章の中でも、この章は最後に読んでください。最初の教訓が「**たいていの場合、これは必要ない**」だからです。スライスや soa 列に対するパイプラインは自動でベクトル化されます ― それが本道であり、幅に依存せず、あなたのデータが走るハードウェアが何であってもそれに追随します。`vecN<T>` は残る 1 ケースのための抜け道です。レーンごとに書き下したい、固定幅のカーネルのための。

## `vecN<T>` ― 値としての SIMD レジスタ

```align
fn main() -> i32 {
    a: vec4<i32> := [1, 2, 3, 4]
    b: vec4<i32> := [10, 20, 30, 40]
    c := a + b          // one instruction: [11, 22, 33, 44]
    d := c * 2          // scalar broadcasts across lanes
    print(d[2])         // 66 — lane read (constant index)
    print(c.max())      // 44 — horizontal reduction
    return 0
}
```

数値スカラーの `vec2` / `vec4` / `vec8` / `vec16` があります。算術はレーン単位で、1 演算あたり 1 命令です。スカラーはブロードキャストされ、`v[i]` はレーンを読み、`mut` なベクトルはレーンへの書き込み(`v[1] = 99`)を許します。要素単位の `a.min(b)` / `a.max(b)`、そして浮動小数の演算 ― `sqrt`・`abs`・`floor`・`ceil`・`round`・`trunc` ― はレーンごとに適用されます。2 つの自由関数が古典的なカーネルをカバーします。`dot(a, b)` と `fma(a, b, c)`(融合積和、丸めは 1 回)です。

整数レーンの意味論はスカラー言語とぴったり一致します。オーバーフローはラップし、ゼロ除算はアボートします ― どのレーンも未定義動作にはなりません。

## マスクと `select` ― 構造からして分岐なし

ベクトルの比較は `maskN<T>` を生みます。これは制御フローとしてではなく、データとして使います。

```align
fn main() -> i32 {
    scores: vec4<i32> := [90, 45, 82, 60]
    m := scores > 80                        // mask4<i32>: [t, f, t, f]
    picked := select(m, scores, scores * 0) // lane-wise blend, no branch
    print(picked.sum_where(m))              // 172 — masked reduction: 90 + 82
    return 0
}
```

`select(mask, a, b)` はレーンをブレンドし、`v.sum_where(mask)` はマスクが残すレーンだけをリダクションします。これは、パイプラインの `where(p)` がコンパイルされる先を明示的に書き下したものです。`xs.where(p).sum()` を分岐なしにするのと同じ identity-select の技です。

## メモリとレジスタを橋渡しする ― `load` / `store`

```align
fn scale_add(xs: slice<f64>, ys: slice<f64>, out dst: slice<f64>) {
    x: vec4<f64> := xs.load(0)      // 4 lanes from the slice, bounds-checked
    y: vec4<f64> := ys.load(0)
    dst.store(0, x * 2.0 + y)       // two instructions, 4 lanes each
}

fn main() -> i32 {
    xs := [1.0, 2.0, 3.0, 4.0]
    ys := [10.0, 10.0, 10.0, 10.0]
    mut out := [0.0, 0.0, 0.0, 0.0]
    mut d: slice<f64> := out
    scale_add(xs, ys, d)
    print(out.sum())                // 60.0
    return 0
}
```

`s.load(i)` は N 個の連続する要素をレジスタに読み込み(N は注釈から)、`s.store(i, v)` は `out` スライス越しにレーンを書き戻します。どちらも境界チェックされます。この組は、手書きのカーネルが配列を歩く方法そのものです ― たいていはスカラーの端数処理を伴う `chunks(N)` の形で。

## `align(N)` ― ロードをアライメントさせたいとき

`align(64) xs := [...]` は配列のストレージを過剰にアライメントし、`align(64) CacheLine { ... }` は構造体を過剰にアライメントします(そして配列のストライドも合わせてパディングします)。証明可能にアライメントされたオフセットでのベクトルロードは、より広いアライメントを使うようになります。FFI で共有するバッファのために `layout(C)` とも組み合わさります。これは現実的だが小さな見返りしかないマイクロ最適化です ― まず計測してください。

## 二層のルール

- **第 1 層(デフォルト): パイプライン。** `xs.map(f).where(p).sum()`、soa 列、`group_by`。幅に依存しません ― コンパイラがターゲットごとにレーン数を選び、同じソースが AVX2 でも NEON でもスケーラブル ISA でもベクトル化されます。この層はあなたのデータに追随します。
- **第 2 層(抜け道): `vecN` / `maskN`。** ドット積・FIR・ブレンドのカーネルで、自分が選んだ固定幅ちょうどのレジスタの振り付けを指定したいとき用です。

第 2 層に手を伸ばすなら、上の `scale_add` のように、境界にスライスを置いた 1 つの小さな関数の中に閉じ込めてください ― 呼び出し側はカーネルが手動ベクトル化されていることを知る必要はありません。そして第 1 層を捨てる前に監査してください。`alignc emit-llvm` は、パイプラインがすでに生成した IR の中の `<4 x i64>` 型を見せてくれます。その監査で最もよくある結末は、手書きのカーネルを削除することです。

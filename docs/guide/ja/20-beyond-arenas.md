# 20. Beyond arenas: pools and lifetimes

> 🌐 [English](../20-beyond-arenas.md) · **日本語**

第5章と第8章で、`arena` がメモリ管理におけるほぼすべての問題を解決する「万能のハンマー」であることを学びました。データが単一の「フェーズ（1つのWebリクエスト、ゲームの1フレーム、コンパイルの1パス）」の間にだけ生きるなら、それを arena に確保し、フェーズが終わったときに建物ごと爆破すればよいのです。

しかし、データが単一のフェーズよりも長生きし、さらに**その寿命が予測不可能なとき**はどうすればよいでしょうか？

マルチプレイヤーゲームのサーバーを想像してください。プレイヤーはランダムな時間にログインし、ログアウトします。フレームごとの `arena` では、プレイヤーのデータがすぐに消えてしまいます。かといって、グローバルな `arena` を1つだけ用意しても、メモリは永遠に解放されず、いずれ枯渇してしまいます。

オブジェクト指向言語であれば、`new Player()` を使い、切断されたときにガベージコレクタ（GC）が後片付けをしてくれるのを待つでしょう。しかし Align では、**データプール（Data Pool）**と**世代付きインデックス（Generational Indices）**を使います。

## プール（The Pool）

プールとは、単なる「あらかじめ確保された並列カラムのブロック（11章のレイアウト）」であり、その行（スロット）を使い回す仕組みです。プレイヤーが参加するたびにOSへメモリを要求するのではなく、連続したカラム — 1スロットにつき1行 — を保持し、空き状況は素の `bool` カラムで自前管理します：

```align
mut alive := [false, false, false, false].to_array()
mut hp    := [0, 0, 0, 0].to_array()
```

プレイヤーが参加したら、最初の空きスロット — `alive[i] == false` — を探し（またはフリーリストを使って `O(1)` で空きを見つけ）、すべてのカラムの行 `i` にデータを書き込んで、そのインデックス（`i64`）を返します。プレイヤーが退出したら、`alive[i]` を `false` に戻すだけです。

OSの呼び出しはゼロ。ガベージコレクタの停止時間もゼロ。ただ配列のバイト列を書き換えているだけです。

## 古びたインデックスの問題

しかし、生の `i64` インデックスをそのまま返すことには、致命的な欠陥があります。
1. アリス（Alice）が参加し、`id = 2` が割り当てられます。
2. ボブ（Bob: アリスの仲間）は、後で彼女を回復するために `target = 2` を保存します。
3. アリスが切断します。スロット2は空きになります。
4. チャーリー（Charlie）が参加し、空いたばかりの `id = 2` が割り当てられます。
5. ボブが `target = 2` に対して回復魔法をかけます。アリスの代わりにチャーリーが回復してしまいます！

これはリソース再利用における古典的な [ABA問題](https://en.wikipedia.org/wiki/ABA_problem) の「古びたハンドル（stale handle）」型です。指していたスロットが、知らないうちに別人に使い回されているのです。もし C++ でポインタを使っていたなら、これは「Use-After-Free（解放後メモリ使用）」という重大なセキュリティ脆弱性になっていたでしょう。

## 世代付きインデックス（Generational Indices）

これを解決するために、単なる `i64` のインデックスを渡すのはやめます。代わりに、インデックスと**世代カウンター（generation）**の両方を含んだ「チケット」を渡すようにします。

```align
Entity { index: i64, generation: i64 }
```

プールには各スロットの世代を追跡するカラムを1本追加し、チケットが今も有効かを確かめる検査を書きます：

```align
mut generation := [0, 0, 0, 0].to_array()
```

```align
fn is_live(alive: slice<bool>, generation: slice<i64>, e: Entity) -> bool =
    alive[e.index] && generation[e.index] == e.generation
```

これで、タイムラインは次のようになります：
1. アリスが参加。スロット2は世代1。アリスには `Entity { index: 2, generation: 1 }` が渡されます。
2. ボブは `target = Entity { index: 2, generation: 1 }` を保存します。
3. アリスが切断。`alive[2]` が `false` になり、**`generation[2]` を 2 にインクリメントします**。
4. チャーリーが参加。スロット2に配置され、`Entity { index: 2, generation: 2 }` が渡されます。
5. ボブが `Entity { index: 2, generation: 1 }` に回復魔法をかけようとします。プールはスロット2を確認し、現在の世代が `2` であり、ボブのチケット（`1`）と一致しないことを発見します。`is_live` は `false` を返し、回復は安全に拒否されます。

## なぜこれが「Align 流」なのか

私たちが達成したことに注目してください：
1. **アロケーションゼロ:** プレイヤーが何百万回参加・退出しても、OSのアロケータは一度も呼ばれません。
2. **キャッシュ局所性:** すべてのプレイヤーはメモリ上に連続して並んでいるため、一括更新（全員に毒ダメージを与える、など）はパイプラインで恐ろしく高速に処理できます — `hp` は最初からカラムなのですから。
3. **絶対的な安全性:** Use-After-Free のバグも、ダングリングポインタも、GCの停止時間もありません。

予測不可能な寿命が必要になったとき、ガベージコレクタを探してはいけません。プールを作り、チケットを配るのです。

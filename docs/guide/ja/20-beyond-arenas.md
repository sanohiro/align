# 20. Beyond arenas: pools and lifetimes

> 🌐 [English](../20-beyond-arenas.md) · **日本語**

第5章と第8章で、`arena` がメモリ管理におけるほぼすべての問題を解決する「万能のハンマー」であることを学びました。データが単一の「フェーズ（1つのWebリクエスト、ゲームの1フレーム、コンパイルの1パス）」の間にだけ生きるなら、それを arena に確保し、フェーズが終わったときに建物ごと爆破すればよいのです。

しかし、データが単一のフェーズよりも長生きし、さらに**その寿命が予測不可能なとき**はどうすればよいでしょうか？

マルチプレイヤーゲームのサーバーを想像してください。プレイヤーはランダムな時間にログインし、ログアウトします。フレームごとの `arena` では、プレイヤーのデータがすぐに消えてしまいます。かといって、グローバルな `arena` を1つだけ用意しても、メモリは永遠に解放されず、いずれ枯渇してしまいます。

オブジェクト指向言語であれば、`new Player()` を使い、切断されたときにガベージコレクタ（GC）が後片付けをしてくれるのを待つでしょう。しかし Align では、**データプール（Data Pool）**と**世代付きインデックス（Generational Indices）**を使います。

## プール（The Pool）

プールとは、単なる「あらかじめ確保された配列（または SoA）」であり、そのスロットを使い回す仕組みです。プレイヤーが参加するたびにOSへメモリを要求するのではなく、連続したメモリブロックを保持し、空き状況を自前で管理します。

```align
Player { name: string, hp: i64 }

Pool {
    slots: array<Option<Player>>,
    next_free: i64,
}
```

プレイヤーが参加したら、最初の `None` のスロットを探し（またはフリーリストを使って `O(1)` で空きを見つけ）、そこに `Player` を入れて、そのインデックス（`i64`）を返します。プレイヤーが退出したら、そのスロットを再び `None` に戻します。

OSの呼び出しはゼロ。ガベージコレクタの停止時間もゼロ。ただ配列のバイト列を書き換えているだけです。

## ABA問題

しかし、生の `i64` インデックスをそのまま返すことには、致命的な欠陥があります。
1. アリス（Alice）が参加し、`id = 4` が割り当てられます。
2. ボブ（Bob: アリスの仲間）は、後で彼女を回復するために `target = 4` を保存します。
3. アリスが切断します。スロット4は `None` になります。
4. チャーリー（Charlie）が参加し、空いたばかりの `id = 4` が割り当てられます。
5. ボブが `target = 4` に対して回復魔法をかけます。アリスの代わりにチャーリーが回復してしまいます！

これはリソースの再利用における古典的な [ABA問題](https://en.wikipedia.org/wiki/ABA_problem) です。もし C++ でポインタを使っていたなら、これは「Use-After-Free（解放後メモリ使用）」という重大なセキュリティ脆弱性になっていたでしょう。

## 世代付きインデックス（Generational Indices）

これを解決するために、単なる `i64` のインデックスを渡すのはやめます。代わりに、インデックスと**世代カウンター（generation）**の両方を含んだ「チケット」を渡すようにします。

```align
Entity {
    index: i32,
    generation: i32,
}
```

そして、プールの各スロットでも世代を追跡できるようにアップグレードします。

```align
Slot {
    value: Option<Player>,
    generation: i32,
}

Pool {
    slots: array<Slot>,
}
```

これで、タイムラインは以下のようになります。
1. アリスが参加します。スロット4の世代は1です。アリスには `Entity { index: 4, generation: 1 }` が渡されます。
2. ボブは `target = Entity { index: 4, generation: 1 }` を保存します。
3. アリスが切断します。スロット4の値は `None` になり、**スロット4の世代を2にインクリメント**します。
4. チャーリーが参加します。彼はスロット4に配置され、`Entity { index: 4, generation: 2 }` が渡されます。
5. ボブが `Entity { index: 4, generation: 1 }` を回復しようとします。プールはスロット4を見ますが、現在の世代が `2` であり、ボブのチケット（`1`）と一致しないことに気づきます。プールは安全にその回復魔法を拒否します。

## なぜこれが Align 流なのか

私たちが何を達成したかに注目してください。
1. **ゼロ・アロケーション:** プレイヤーが何百万回参加・退出を繰り返しても、OSのアロケータへの呼び出しは一度も発生しません。
2. **キャッシュ・ロカリティ:** すべてのプレイヤーはメモリ上に連続して存在しているため、パイプラインを使った一括更新（全員に毒ダメージを与えるなど）が信じられないほど高速になります。
3. **絶対的な安全性:** Use-After-Free のバグも、ダングリングポインタも、ガベージコレクションによる停止もありません。

寿命が予測不可能なデータが必要になったとき、ガベージコレクタを探してはいけません。プールを作り、チケットを配るのです。

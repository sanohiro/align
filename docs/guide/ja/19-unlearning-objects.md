# 19 — オブジェクト指向のアンラーニング (Unlearning objects)

> 🌐 [English](../19-unlearning-objects.md) · **日本語**

もしあなたが Java や C#、Python といった言語から Align にやってきたなら、「世界を『相互に作用するオブジェクトのグラフ』としてモデル化しようとする直感」が働くはずです。しかし Align では、その直感はあらゆる場面で言語の思想と衝突します。Align にはオブジェクトも、クラスも、継承もありません。

この章はパラダイムシフトのためのクックブックです。オブジェクト指向でよくある問題解決のアプローチを、どのように「Align の流儀（データ指向）」へと変換すべきかを解説します。

## 1. 「状態を持つエンティティ」というアンチパターン

**OOP のアプローチ:** 
自身の `health`（体力）や `x`, `y` 座標を持ち、自らの状態を更新する `update()` メソッドを備えた `Player` クラスを作ります。

**Align のアプローチ:**
データと振る舞いは完全に分離します。さらに言えば、個別の `Player` 1つ1つは、多くの場合「正しい抽象化の単位」ではありません。あなたは「ある1人のプレイヤーを更新する」のではなく、「すべての座標を更新する」のです。

```align
// Player { health, x, y } という「個体」で考えるのをやめる
arena {
    players: soa<Player> = load_players()
    
    // すべての座標を、キャッシュフレンドリーな1回のパスで一括更新する
    players.x.zip(players.velocity_x).map(fn (x, v) { x + v }).to_array()
}
```

## 2. 「ポリモーフィックなリスト」というアンチパターン

**OOP のアプローチ:** 
`Shape` インターフェースのリストに、`Circle` や `Rectangle` といったオブジェクトを混ぜて格納します。ループを回し、各要素に対して `shape.draw()` を呼び出します。これにより、ループのたびに仮想メソッドディスパッチ（そしてキャッシュミス）が発生します。

**Align のアプローチ:**
要素の数が少なく種類が混ざっているなら、直和型（`enum`）を使います。あるいは処理速度を最優先するなら、配列そのものを型ごとに分離します。

もし混ぜて保持する必要があるなら：
```align
enum Shape {
    Circle { radius: f32 },
    Rect { w: f32, h: f32 },
}

shapes.map(fn s {
    match s {
        Circle { radius } => draw_circle(radius),
        Rect { w, h } => draw_rect(w, h),
    }
})
```
しかし、真のデータ指向のアプローチは、すべての Circle を1つの `soa<Circle>` に、すべての Rect を `soa<Rect>` に保存し、**一切の分岐を持たない2つの独立した爆速のパイプラインで一気に処理する**ことです。

## 3. 「見えないアロケーション」というアンチパターン

**OOP のアプローチ:** 
ループの中でリストに要素を `append` していきます。リストは自動的に自身のサイズを拡張し、予測不可能なタイミングでヒープメモリを確保（アロケーション）します。

**Align のアプローチ:**
Align にも `heap.alloc` はありますが、それが自然に使われることは稀です。動的なメモリが必要なときは `arena` を使います。arena ブロックが終了した瞬間、すべてのメモリは一瞬で解放されます。ホットループの中で個々のオブジェクトを `new` したり `delete` したりすることは決してありません。

```align
arena {
    // 途中で個別に free することなく、arena に結果を蓄積していく
    mut results := []
    lines.where(.is_error).map(fn l { results.push(l.msg) })
} // ズドン。すべて消え去る。
```

## 4. 「Getter/Setter」というアンチパターン

**OOP のアプローチ:** 
状態をカプセル化し、振る舞いを注入するために、フィールドを `get_health()` と `set_health()` の背後に隠蔽します。

**Align のアプローチ:**
データはただのデータです。構造体（Struct）のフィールドは公開されています。もしデータを変換する必要があるなら、構造体を受け取って新しい値を返す「純粋な関数」を書きます。Alignの「隠し事をしない（Nothing hidden）」という哲学は、単にメモリアドレスを読み取るためだけに任意のコードを隠れて実行させないことを意味します。

## まとめ

オブジェクト指向のアンラーニングとは、「そのアクションを『実行する主体（オブジェクト）』は誰か？」を探すのをやめ、**「『データ』はどのように流れるか？」** を考え始めることです。データを平らに敷き詰め、パイプラインに流し込んだとき、マシンはあなたが想像するよりも遥かに速くコードを実行してくれます。

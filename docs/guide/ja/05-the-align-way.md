# Align の流儀

> 🌐 [English](../05-the-align-way.md) · **日本語**

Align のコードを速く、そして慣用的(イディオマティック)にするイディオムを短くまとめたものです — いわば「文化」であり、経験を積んだ Align プログラマがものごとに手を伸ばすときのやり方です。それぞれは、これまでの章で紹介してきた習慣であり、ここではチェックリストとしてまとめます。

## 変換を記述する、反復(イテレート)しない

ループに手を伸ばす前に `map`/`where`/`reduce` に手を伸ばしてください。パイプラインは一度のパスに融合されますが、一時変数を使った手書きのループはそうなりません。`for` を書いているなら、その処理をパイプラインの語彙で言えないかを問うてください — たいていの場合、言えます。

```align
// idiomatic
total := xs.map(f).where(p).sum()

// avoid, when a pipeline expresses it
// mut total := 0
// for x in xs { ... }
```

## エラーは分岐ではなく `?` で処理する

失敗しうる関数は `Result` を返します。それを `?` を使って呼び出してください。ハッピーパス(正常系)は上から下へと読め、エラーパスはコンパイラがホットコードから離れた場所に配置する cold edge(コールドエッジ)になります。

```align
import std.fs

fn load(path: str) -> Result<Config, Error> {
    raw := fs.read_file(path)?
    cfg := parse(raw)?
    return Ok(cfg)
}
```

## ライフタイムを選び、あとはメモリのことを忘れる

ローカルな値 → ただの value でよい。大量に allocate するフェーズ → `arena {}` で包む。その一度の判断さえすれば、あとはコンパイラがやってくれます。`free` もなく、リークもなく、ダングリングもありません。

## まとめて扱うデータは SoA でレイアウトする

コレクションを繰り返し処理するなら `soa<T>` を使ってください。キャッシュ密度とベクトル化は、そのレイアウトから自然に得られます。AoS は、丸ごと、たまにしか触らないデータのためのものです。

## コンパイラに「形」を見せる

Align の速さは、コンパイラがものごとを証明できることに由来します — 連続したメモリ、エイリアシングがないこと、非 null であること、cold error path、arena のライフタイム。ありとあらゆる制約 — null がないこと、エラーモデルがひとつであること、Move 型、推論された region — は、あなたが注釈を書くこと*なしに*コンパイラがそれらの性質を推論できるようにするために存在します。素直な方向で書くこと(ループよりパイプライン、番兵値より `Result`、散らばった allocation より arena)こそが、その推論を生かし続けるものです。慣用的な Align は、そのまま速い Align です — この2つは設計上、同じものなのです。

## 何も隠さない

ある行が allocate しているなら、それは見てわかります(`heap.new`、`arena` ブロック、`string` ビルダー)。失敗しうる処理は `Result` を返します。並列に走る処理なら、あなたが `par_map` か `task_group` を書いたはずです。探し回るべき隠れたコストは存在しません — これこそが、一度書かれた Align のコードを信頼しやすいものにしているのです。

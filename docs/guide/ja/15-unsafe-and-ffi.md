# 境界: unsafe と C FFI

> 🌐 [English](../15-unsafe-and-ffi.md) · **日本語**

これまでに説明してきたすべての安全性保証（ダングリングビューが発生しない、二重解放（Double Free）が起きない、`par_map` でデータ競合が発生しないなど）は、コンパイラがコードの挙動をすべて見通すことができるからこそ成り立っています。しかし「世界の端」――つまり C ライブラリとの連携や、手動で管理される生メモリバッファなどを扱う領域――では、コンパイラには限界があります。この問題に対する Align のアプローチは標準的でありながら、対象範囲がごく小さく保たれています。`unsafe {}` ブロックを使用することで、「ここから先はプログラマが安全性の保証に責任を持つ」という箇所を正確にマークし、そのブロックの外側からは誰も安全性を壊せないように隔離します。

## `unsafe {}` と `raw.*`

`raw` は生ポインタを表す型です。これに触れる手段は5つの `raw.*` 操作に限られており、しかもそれらは **`unsafe` ブロックの内側でのみ** 呼び出しが許可されています。

```align
fn main() -> i32 {
    unsafe {
        p := raw.alloc(16)          // 16 raw bytes
        raw.store(p, 0, 42)         // write an i64 at byte offset 0
        raw.store(p, 8, 99)
        a: i64 := raw.load(p, 0)    // read back — type from the annotation
        b: i64 := raw.load(p, 8)
        raw.free(p)                 // yours to free — a raw is never dropped
        print(a + b)                // 141
        return 0
    }
}
```

`alloc`、`free`、`load`、`store`、`offset`。これが Align における `unsafe` 操作の全語彙です。C のような暗黙のポインタ演算子（`*p` や `p++`）もなければ、ポインタを介した複雑なキャスト構文も存在しません。コードベースを `grep` で簡単に検索できる、これら5つの名前付きの操作関数があるだけです。

なお、`raw` ポインタそのものを**保持すること**自体は安全とみなされます（`raw` は Copy 値なので、関数間で自由に受け渡しが可能です）。`unsafe` ブロックが必要になるのは、実際にそのポインタが指すメモリを**操作（読み書き・解放など）**する瞬間だけです。

`unsafe` が**しないこと**を理解するのも重要です。これは「コンパイラのチェックをすべてオフにする魔法のスイッチ（モード切り替え）」ではなく、単なる「マーカー」です。`unsafe` ブロックの内側であっても、アリーナからのエスケープ検査、Move セマンティクスの検査、通常のスライス等に対する境界検査は引き続きすべて有効に働きます。また、[10](10-closures-and-parallelism.md) 章で触れた純粋性推論は、`unsafe` ブロックを含む関数を自動的に「非純粋（impure）」と判定するため、生メモリを操作するコードが誤って `par_map` のデータ並列処理に乗ってしまうことは絶対にありません。

## `extern "C"` — 外の世界を宣言する

```align
extern "C" {
    fn abs(x: i32) -> i32
    fn labs(x: i64) -> i64
}

fn main() -> i32 {
    unsafe {
        print(abs(-7))      // 7 — a real libc call
        print(labs(-40))    // 40
        return 0
    }
}
```

連携したい C 言語の関数のシグネチャ（型定義）を宣言し、それを `unsafe` ブロックの内側で呼び出します。（コンパイラは C 側の関数が実際に何をするかを検査できないため、このブロックは「プログラマであるあなたが安全性を確認した」とコンパイラに宣言するためのものです）。
標準的な `libc` と `libm` は自動的にリンク解決されます。それ以外の外部ライブラリを使用する場合は、`link` 属性でライブラリ名を明示的に指定します。

```align
extern "C" link("m") {
    fn sqrt(x: f64) -> f64
    fn cbrt(x: f64) -> f64
}
```

## データを渡す

スカラー型は C の型と直接対応します（`i32` ↔ `int32_t`、`f64` ↔ `double` など）。一方、Align のビュー型（`str`、`slice<T>`、`bytes`）を C の関数に渡す場合、それらは自動的に**内部のデータポインタ**（C 言語のポインタ）へと低下（デグレード）して渡されます。ビューの「長さ（要素数やバイト数）」は暗黙的には渡されないため、引数として自分で明示的に渡す必要があります。

```align
extern "C" fn write(fd: i32, buf: str, count: i64) -> i64

fn main() -> i32 {
    msg := "written by libc\n"
    unsafe {
        n := write(1, msg, msg.len())   // fd 1 = stdout
        print(n)                        // 16
        return 0
    }
}
```

**FFI を扱う際に絶対に忘れてはならない（どこかに刺青しておくべき）唯一のルール：**
Align の文字列は C 言語のような NUL（`\0`）終端文字列ではありません。したがって、`write`、`memcmp`、`memcpy` のような「長さを引数に取る C API」に渡すのは安全ですが、`strlen` や `printf("%s")` のような「NUL 終端を期待する C API」にそのまま渡すと、メモリの末尾を超えてバッファオーバーリードを引き起こします。

また、C 側に構造体を渡す必要がある場合は、`layout(C)` 修飾子を使ってメモリレイアウトを C 言語の ABI に固定します。これにより、フィールドの宣言順序が維持され、C と同じアライメント規則が適用され、コンパイラによる勝手なフィールドの並べ替えが禁止されます（この指定がないと、Align はメモリ密度の最適化のためにフィールドの順序を自動的に並べ替える可能性があります）。

```align
layout(C) Point { x: i32, y: i32 }      // matches `struct { int32_t x, y; }`
```

`layout(C)` が指定された構造体は、ポインタ経由（`raw` を通して）、あるいは**値渡し**によって言語の境界を越えることができます（現在の実装では SysV x86-64 ABI に従い、サイズが 16 バイト以下のレジスタクラス構造体の値渡しに対応しており、clang の挙動と正確に一致します。それより大きな構造体の値渡しは現在実装中です）。

C 側が所有権を持つメモリを受け取る場合、それは `raw` ポインタとして Align 側に戻ってきます。C のポインタには長さ情報が含まれていないため、Align 側で勝手に安全なビュー（スライスなど）を装うことはしません。受け取ったメモリは `raw.load` を使って手動で読み取るか、別の引数などで得た「正しい長さ情報」と組み合わせてから、安全なビューとしてラップし直す必要があります。

## 規律

言語間の境界はできる限り薄く、かつ監査（レビュー）が容易な状態に保つべきです。理想的な設計は、1つの専用モジュールだけが `extern` 宣言と `unsafe` ブロックを隠し持ち、その境界の内部でデータの変換（ビューの長さを引き渡す、`raw` ポインタを扱って適切に解放する、C のエラーを `Result` に変換するなど）を済ませた上で、外部の Align コードに対しては**完全に安全な API のみをエクスポートする**ことです。

これにより、そのモジュールを利用する呼び出し側のコードは、純粋な Align コードと全く同じ安全性の保証を得ることができます。`unsafe` ブロックの外側では、コンパイラの健全性（Soundness）が破られることはあり得ないからです。`grep unsafe` コマンドの結果こそが監査すべきサーフェス（表面）であり、健全に設計されたプログラムであれば、その結果はコードベース全体に散らばるのではなく、画面1ページに収まる程度の分量になるはずです。

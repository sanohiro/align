# ジェネリクスとモジュール

> 🌐 [English](../09-generics-and-modules.md) · **日本語**

多くのプログラミング言語では数章にわたって解説されるような機能ですが、Align では意図的にシンプルに保たれています。ジェネリクスは、たった3つの組み込み境界（バウンド）と完全な型推論によって提供され、モジュールシステムは単なる「ファイル」として扱われます。

## ジェネリック関数

```align
fn max<T: Ord>(a: T, b: T) -> T = if a > b { a } else { b }
fn add<T: Num>(a: T, b: T) -> T = a + b
fn same<T: Eq>(a: T, b: T) -> bool = a == b
fn unwrap_or<T>(o: Option<T>, fallback: T) -> T = o else fallback

fn main() -> i32 {
    print(max(7, 12))       // 12   — T = i64, inferred
    print(max(1.5, 0.5))    // 1.5  — T = f64
    print(add(40, 2))       // 42
    print(same("a", "a"))   // true
    print(unwrap_or(Some(5), 0))    // 5
    return 0
}
```

型パラメータは**常に引数から推論されます**。Rust の turbofish (`::<>`) のような構文もなく、覚える必要のある明示的なインスタンス化の構文は存在しません。ジェネリクスはモノモーフィゼーション（単相化）されます。具体型が使用されるごとに、プログラマが手書きしたかのように専用のコードが生成されてコンパイルされます。そのため、実行時ディスパッチのオーバーヘッドはゼロです。

## 境界: `Num` ⊃ `Ord` ⊃ `Eq`

境界（トレイト境界に相当）が指定されていない `T` は、不透明（opaque）な型として扱われます。値のムーブ、保存、戻り値としての返却は可能ですが、それ以外の操作は何もできません。型に対する操作能力は、以下の3つの組み込み境界によってのみ与えられます。

- `T: Eq` ― `==`、`!=`
- `T: Ord` ― 比較(`Eq` を含意)
- `T: Num` ― 算術(`Ord` を含意)

境界によって許可されていない操作を使用すると、**関数を定義した時点で**コンパイルエラーになります。呼び出し側でエラーになったり、後になって他の開発者のビルド環境で突然失敗したりすることはありません。

これが Align の型制約システムのすべてであり、これは意図的な設計です。**ユーザー定義のトレイトやインターフェースは存在しません**（これは機能の欠落ではなく、明確な意思決定です）。複雑なトレイト階層は、チューリング完全な型レベル言語という「第2の言語」を生み出す温床になります。人間はそれを読み飛ばそうとし、AI は存在しない型を幻覚（ハルシネーション）してしまいます。Align の設計上の賭けは、「3つの組み込み境界と具体型があれば、データ指向プログラミングにおける現実的なジェネリクスのユースケースは十分にカバーでき、それ以外の複雑な処理は素朴な関数として記述した方が良い」というものです。

## ジェネリック型

構造体と直和型（Sum type）も同様に型パラメータを取ることができ、その型は値の構築時に推論されます。

```align
Pair<T> { a: T, b: T }

Opt<T> { Has(T), Empty }

fn sum_ints(p: Pair<i64>) -> i64 = p.a + p.b

fn main() -> i32 {
    p := Pair { a: 40, b: 2 }       // Pair<i64>, inferred
    q := Pair { a: 1.5, b: 2.5 }    // Pair<f64>
    print(sum_ints(p))              // 42
    print(q.a + q.b)                // 4.0
    o := Opt.Has(9)                 // Opt<i64>, inferred from the payload
    v := match o {
        Has(n) => n,
        Empty  => 0,
    }
    print(v)                        // 9
    return 0
}
```

`Option<T>` と `Result<T, E>` もまさにこの仕組みを利用して、言語に組み込みで提供されています。現時点での実装上の制限についても正直に触れておきます。ジェネリックな構造体を受け取るジェネリックな関数（例：`fn first<T>(p: Pair<T>) -> T`）は現在実装中です。また、ペイロードを持たないバリアントを単独で構築する場合（例：`Opt.Empty`）、型 `T` を推論・確定するための文脈（変数への代入など）が必要になります。

## モジュールはファイル

1つのファイルが1つのモジュールに対応し、`module` 宣言の名前はファイル名と一致している必要があります。`import` 文は、同じディレクトリにある兄弟ファイルを読み込みます。`pub` キーワードを付けない限り、宣言はすべてプライベート（private）になり、モジュールをまたぐ参照は常にモジュール名で修飾する必要があります。ヘッダーファイルも、マニフェストも、複雑な検索パスの設定も不要です。

```align
// geom.align
module geom

pub Point { x: i64, y: i64 }
pub SCALE: i64 := 3
pub fn area(p: Point) -> i64 = p.x * p.y

fn hidden(x: i64) -> i64 = x        // private: invisible to importers
```

```align
// main.align
module main

import geom

fn main() -> i32 {
    p := geom.Point { x: 4, y: 5 }
    print(geom.area(p) * geom.SCALE)    // 60
    return 0
}
```

`alignc run main.align` を実行すると、コンパイラはエントリファイルの隣にある `geom.align` を自動的に見つけます。`import util.math` と書けば、`util/math.align` ファイルが読み込まれます。名前の修飾規則は絶対です。インポートした型は `geom.Point` のように修飾して使用する必要があり、修飾なしの `Point` として使うことはできません。インポートした直和型のバリアントも `geom.Color.Red` のようになります。コード上のどの名前を見ても、それがどのファイルから来たものかが正確にわかります。ファイル冒頭の `import` リストをさかのぼって探すような「考古学」は不要です。モジュールのエイリアス（`import x as y`）やグロブインポート（ワイルドカード）もありません。対象を参照する方法は常に1つだけです。

`import` キーワードは、組み込みの機能モジュールを使用する際にも用います（例：`import std.fs`、`import core.json`）。これは、そのファイルが「外界のどのリソースにアクセスするか」をファイルの冒頭で宣言する役割を果たします。逆に言えば、`std` のインポートが1つもないファイルは、I/O 操作を一切行わないことが保証されます。[13](13-std-os.md) 章で解説する標準ライブラリの設計は、この考え方の上に成り立っています。

## プログラムのかたち

小さなプログラムは1つのファイルで完結します。プログラムが成長すると、データの境界がモジュールの分割線になります。データ構造（レコード型）とそれに紐づく処理は別のモジュール（例えば `records.align`）へ切り出され、I/O などの外界との境界部分は `main.align` に残ります。そして、`pub` キーワードによって公開インターフェースが明示されます。参照には常にモジュール名が修飾され、可視性も明示的であるため、モジュールの本当のインターフェースは単純な `grep` で簡単に把握できます。`pub` と書かれた行そのものが、モジュールの契約（API）となるのです。
